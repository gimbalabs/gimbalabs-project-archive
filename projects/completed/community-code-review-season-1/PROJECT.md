# Community Code Review Season 1

"Mob Family Code Review" - "DatumDons"

## Gimbalabs Primary Driver
To provide open, replicable, safe spaces to learn, to explore, and to empower individuals and organizations anywhere so that we can solve meaningful problems

## Proposal Driver
The current situation is that reading and understanding smart contract code remains a major challenge for aspiring developers in the Cardano ecosystem. Many open-source projects lack informal review or documentation before deployment, and formal audits are often too expensive for small teams.

This creates barriers to entry for learners, increases deployment risks for builders, and limits collaboration across the ecosystem.

By offering a learner-first, community-based code review series, we aim to:
- Improve accessibility and comprehension of validator code
- Build developer skills and confidence
- Offer valuable feedback to builders before launch

This proposal supports Gimbalabs' mission by creating an open, replicable safe space where anyone — regardless of experience level — can learn by doing, ask questions, and contribute to the improvement of real-world blockchain projects.

## Proposal

### 1. Title or Brief Description
Community Code Review: A learner-first series of Aiken validator walkthroughs

### 2. Background or Rationale
1. Reading smart contract code is a critical skill for Cardano developers and power users, yet it remains a major hurdle for many newcomers.
2. Formal audits are prohibitively expensive for small teams and experimental dApps, making informal but structured reviews valuable.
3. Deploying unreviewed contracts to mainnet poses reputational and security risks that could be mitigated through peer validation.
4. Gimbalabs has a track record of effective peer-based learning using methods like Pair Programming and Mob Coding.
5. Community Code Review was piloted with Codecow's Double Spend validator, demonstrating potential as a repeatable process.
6. There's a growing number of open-source Aiken projects that would benefit from external review before launch.
7. Junior developers often lack structured, low-risk opportunities to read and critique real code alongside experts.
8. Community-based reviews provide a space for "noob" learners to gain practical experience while also offering value to project builders.
9. Reviewing high-profile or "cared about" projects can help establish credibility, attract interest, and bring visibility to lesser-known work.
10. This initiative aligns with Gimbalabs' mission by creating an open, replicable, and empowering learning format for decentralized development.

### 3. Intended Outcome
- Support open-source, small-scale, and experimental projects by creating opportunities for beginner developers to learn, contribute, and upskill.
- Run a recurring series of structured Community Code Review sessions based on a learner-first, "mob coding" format.
- Establish reputation and relationships across the Cardano ecosystem by starting with high-profile projects and expanding to lesser-known work.
- Help identify code quality issues and security concerns prior to mainnet deployment.
- Define and document best practices for informal community code reviews.
- Model a learner-first environment where novice questions and misunderstandings are welcomed as part of the review process.
- Enable aspiring developers (e.g. Newman, Harsha, CodeCow) to lead sessions, supported by experienced developers, as a scaffolded audit-learning experience.
- Ensure that the reviewing "noob" prepares in advance, building context and familiarity before leading the session.
- Document and highlight important open-source projects for the broader Cardano community.
- Make code review a fun and meaningful way to onboard new developers and build community.

### 4. Details of the Proposal

**Short Description:**
This project will host a series of 90-minute Community Code Review sessions focused on open-source Aiken smart contract repositories, using a "mob coding" approach.

**Tag line:**
"By the noobs, for the noobs" - a learner-first audit process.

**Roles:**
- *Facilitator* – organizes and keeps the session on track
- *Builder* – presents and answers questions about their smart contract code
- *Noob / Junior Dev* – shares screen, reads code aloud, asks questions
- *Community Experts* – answer questions, offer commentary and guidance

**Method:**
The junior dev "drives" the code walkthrough, reading the validator logic out loud and asking questions as they go. The builder responds and reflects on design decisions. Senior devs provide commentary and fill in context. This format turns code review into a shared learning experience.

**Tasks:**
- Create a reusable routine and guide for running sessions
- Promote the series and recruit projects to participate
- Announce meetings, confirm roles, and coordinate scheduling
- Facilitate and record each session
- Upload videos to YouTube with descriptions and timestamps
- Document highlights and learnings from each session
- Encourage post-session discussion and follow-up contributions

**Phases:**
1. **Pilot:** Internal soft launch with "known and cared about" projects
2. **Document:** Refine role clarity, methods, and capture learnings in a video series
3. **Open Call:** Invite new projects to apply for review sessions
4. **Sustain:** Build a rhythm of recurring sessions, rotating hosts and participants

**Example Projects:**
- The Aiken getting started validators
- The Andamio validators
- Asteria by TxPipe and Asteria-GL (Gimbalabs fork)
- Lenfi
- Yamfore

### 10. Potential Risks and Mitigation Strategies

**Risk: The noob is underprepared or too inexperienced**
Mitigation: The noob is defined as someone taking the AikenPBL or equivalent experience. If they are too inexperienced we swap them with a different one during the session.

**Risk: The review lacks the rigor of a professional audit**
Mitigation: Frame as a "pre-audit" learning step, not a replacement; focus on comprehension and transparency, not certification. It's not a rigorous audit. It's a 'read-through' for educational purposes.

**Risk: Low attendance or weak community engagement**
Mitigation: This proposal process serves to validate community interest. We can avoid rigid scheduling; only run sessions when a group is motivated; promote widely and emphasize value for all roles.

**Risk: Too much administrative overhead**
Mitigation: Use templates and shared docs to reduce manual work; keep logistics lean; focus on what matters — the builder-noob interaction and the learning experience.

### 11. Review and Adaptation Plan
- After each session, hold a short retro with core contributors (noob, builder, facilitator) to discuss what worked and what to improve
- Collect feedback via short forms from all attendees (live or async)
- Log key takeaways, friction points, and ideas in a shared Notion or document
- Use these insights to refine the meeting template, checklist, and onboarding guide
- Publish learnings in a public changelog or "season summary"
- At the end of the first season (3 months), assess whether to continue, expand, or pause the initiative

### Don for DatumDons
Leads the weekly code review session by guiding discussion around a chosen Aiken validator contract.

**Core Tasks:**
- Select and prepare a validator contract for the session
- Review and understand the code in advance
- Guide the conversation during the session, highlighting key patterns, design choices, and questions
- Actively engage with Capos and guests to encourage meaningful discussion and shared learning

### Capo for DatumDons
Capos support the Don by deeply engaging with the code under review—asking questions, offering insights, and challenging assumptions.

**Core Tasks:**
- Ask clarifying and challenging questions to promote shared understanding
- Offer alternative interpretations or improvements
- Help the Don and each other explore the deeper logic and structure of the code
- Foster thoughtful debate and collaborative learning
