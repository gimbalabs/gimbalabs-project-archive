## Classifications

### Module 1: What We Built

#### SLT 1.1: "I can categorize Gimbalabs' completed and active projects by theme and structure."
- **Lesson Type**: Exploration
- **Key Factors**: Analytical work on the project archive. No UI or procedure — learners develop their own taxonomy through examination.
- **Edge Notes**: Could feel like Organization Onboarding since it's org-specific, but the skill is analytical categorization, not learning org procedures.
- **Inputs Needed**: Project archive data (this repo), framing questions about what counts as a "category"

#### SLT 1.2: "I can identify recurring patterns across Gimbalabs projects."
- **Lesson Type**: Exploration
- **Key Factors**: Pattern recognition across a body of work. Open-ended — different learners may surface different patterns.
- **Inputs Needed**: Project archive data, prompts to look beyond obvious groupings

### Module 2: What's Missing

#### SLT 2.1: "I can compare Gimbalabs' project portfolio against our primary driver."
- **Lesson Type**: Exploration
- **Key Factors**: Compare is a strong Exploration verb. Requires learners to hold the primary driver ("open, replicable safe spaces to learn, explore, and empower") alongside the actual project history and find the gaps.
- **Inputs Needed**: Primary driver statement, project portfolio summary, framing questions about alignment

#### SLT 2.2: "I can assess which project categories are over-represented or absent from our history."
- **Lesson Type**: Exploration
- **Key Factors**: Assess implies evaluation and judgment. Builds directly on 1.1's categorization.
- **Inputs Needed**: Categorization from Module 1, ecosystem context for what "absent" might mean

### Module 3: Unfinished Business & Unrealized Potential

#### SLT 3.1: "I can trace the evolution of a Gimbalabs project across funding rounds."
- **Lesson Type**: Exploration
- **Key Factors**: Historical analysis. Learners examine how ideas morphed across F11, F12, F13, F14 and Intersect grants. The "tracing" is interpretive, not procedural.
- **Inputs Needed**: Funding history, project lineage examples (e.g., Dandelion → Dandelion-lite, Which Spaces across rounds, Sustain+Maintain continuity)

#### SLT 3.2: "I can distinguish between projects that ended because they were complete and projects that ended because resources ran out."
- **Lesson Type**: Exploration
- **Key Factors**: Evaluative judgment about project endings. No right answer for many cases — learners develop perspective.
- **Inputs Needed**: Completed project archive, framing questions about what "done" means vs. "stopped"

### Module 4: Beyond Catalyst

#### SLT 4.1: "I can map Gimbalabs' funding history and its concentration risks."
- **Lesson Type**: Exploration
- **Key Factors**: Analytical — learners construct a funding picture and evaluate risk. The "mapping" is conceptual, not cartographic.
- **Inputs Needed**: Budget data from project files, Catalyst funding totals, treasury info

#### SLT 4.2: "I can propose alternative sustainability models for a Gimbalabs project that don't depend on Project Catalyst."
- **Lesson Type**: Exploration
- **Key Factors**: Open-ended brainstorming and creative thinking. No known framework to follow — learners generate possibilities.
- **Inputs Needed**: Framing questions about sustainability, examples from other open-source/community projects

#### SLT 4.3: "I can design a project that generates value for the community without requiring external funding to start."
- **Lesson Type**: Exploration
- **Key Factors**: Creative design challenge. Conceptual — the output is an idea with a rationale, not a full spec.
- **Inputs Needed**: Constraint framing ("no external funding"), examples of bootstrapped community projects

### Module 5: What the Ecosystem Needs That Nobody's Building

#### SLT 5.1: "I can identify unsolved problems in the Cardano ecosystem that Gimbalabs is positioned to address."
- **Lesson Type**: Exploration
- **Key Factors**: Requires ecosystem awareness and honest self-assessment of community capabilities. Outward-looking.
- **Inputs Needed**: Ecosystem landscape framing, community capability inventory from earlier modules

#### SLT 5.2: "I can evaluate a project idea against Gimbalabs' primary driver and the community's demonstrated strengths."
- **Lesson Type**: Exploration
- **Key Factors**: Evaluate is the highest Bloom's level in the set. Learners apply a multi-criteria judgment framework.
- **Inputs Needed**: Primary driver, strengths inventory, candidate project ideas from 5.1

### Module 6: Designing the Next 25

#### SLT 6.1: "I can draft a project proposal using the 11-point framework."
- **Lesson Type**: How To Guide
- **Key Factors**: There is a literal template (proposals/TEMPLATE.md) with 11 defined points. The learner follows a specific structure. Most procedural SLT in the course.
- **Inputs Needed**: Proposal template, example proposals from the archive, a project idea from Modules 4-5

#### SLT 6.2: "I can scope a project for a small team that can ship something meaningful within 3 months."
- **Lesson Type**: Exploration
- **Key Factors**: Scoping requires judgment about what's achievable — not a checklist. Learners develop intuition about team capacity and deliverable sizing.
- **Inputs Needed**: Examples of well-scoped vs. over-scoped projects from the archive, team size data

#### SLT 6.3: "I can sequence a set of projects so that early projects create momentum that later projects depend on."
- **Lesson Type**: Exploration
- **Key Factors**: Strategic thinking about dependencies and momentum. The "set of projects" is the portfolio they've been generating throughout the course.
- **Inputs Needed**: Project ideas from earlier modules, framing about dependencies and sequencing

### Module 7: Building in Public (Optional)

#### SLT 7.1: "I can design a project documentation strategy that makes our work legible and replicable by communities outside Gimbalabs."
- **Lesson Type**: Exploration
- **Key Factors**: Design thinking about communication and replicability. No standard procedure — learners develop their own strategy.
- **Inputs Needed**: Examples of well-documented open-source projects, Gimbalabs' current documentation practices

#### SLT 7.2: "I can define success metrics for a project that measure community impact beyond milestone delivery."
- **Lesson Type**: Exploration
- **Key Factors**: Defining metrics requires perspective on what matters. Challenges the default "did we hit milestones" thinking.
- **Inputs Needed**: Current evaluation criteria from project files, examples of impact metrics from other communities

## Heuristics Developed

### Verb Patterns
- SLTs with "categorize", "identify", "compare", "assess", "trace", "distinguish", "map", "propose", "design", "evaluate", "define" → **Exploration** in reflective/generative courses. These verbs all involve developing judgment, not following procedures.
- SLTs with "draft" + a named template/framework → **How To Guide** when the template already exists and the learner follows its structure.

### Subject Matter Patterns
- Retrospective analysis of an organization's own work → **Exploration**. The learning is in the interpretation, not the mechanics.
- Ecosystem scanning and gap identification → **Exploration**. No procedure exists — learners generate insight.
- Template-driven document creation → **How To Guide**. The procedure is the template.

### Ambiguity Patterns
- When an SLT says "design" or "propose" something new, it could be Exploration (conceptual) or How To Guide (if there's a design process). Resolved by asking: "Is there an existing framework to follow, or is the learner generating something new?" If generating → Exploration.
